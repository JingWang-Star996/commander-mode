# 指挥官模式 (Commander Mode)

## ⚠️ 最高优先级 - 强制执行

**来源**: 王鲸 2026-04-09 22:14 明确要求
**原因**: 主会话执行重任务导致响应极慢，严重影响开发效率
**API 并发**: 支持 5 个 agent 并行
**修复事故**: INC-2026-0409-007 — 主会话先干活再回复，用户等待过久

---

## 🔴 P0 规则：先回复，再分发

**问题根源**：LLM 默认行为是「思考/执行 → 输出」，用户感知为「等很久才收到回复」。
**解决方案**：强制拆分为「输出 → 思考/执行 → 输出」。

### 收到任务后的强制流程

```
用户发消息
    ↓ (3 秒内)
① 主会话：立即回复用户
    "收到，正在处理。"
    或 "收到，[简要复述任务]，预计 X 分钟完成。"
    ↓ (同时)
② 主会话：sessions_spawn 子代理去执行
    ↓
③ 子代理：执行任务
    ↓
④ 子代理：完成 → 自动汇报结果
    ↓
⑤ 主会话：转发结果给用户
```

### 硬性要求

| 规则 | 要求 | 违反后果 |
|------|------|----------|
| **回复时限** | 收到任务后 **3 秒内** 必须回复用户 | 记录为 INC 事故 |
| **回复内容** | 至少包含：确认收到 + 简要任务描述 | 视为无效回复 |
| **回复顺序** | 回复用户 → 再 spawn 子代理 | 顺序颠倒 = 严重违规 |
| **禁止沉默执行** | 绝不先做工具调用再回复 | 视为违反指挥官模式 |

### 回复模板

```markdown
# 简单任务
收到，正在处理。

# 中等任务
收到，正在处理 [任务简述]，预计 X 分钟完成。

# 复杂任务（多子代理）
收到，任务已拆解为 X 个子任务，正在并行处理中：
- 子任务 A：[简述]
- 子任务 B：[简述]
完成后汇总结果。
```

### ❌ 绝对禁止（违反 = INC 事故）

1. ❌ 收到任务后先执行工具调用，完成后才回复用户
2. ❌ 沉默执行超过 3 秒不回复
3. ❌ 回复中说"完成后再告诉你"之类的等待话术
4. ❌ 把"回复用户"放在子代理完成之后

## 🔴 P0 规则：Agent 空闲计数器

**来源**: 王鲸 2026-04-10 01:38 明确要求
**原因**: 主会话需要感知 agent 池空闲状态，避免空闲时自己干活而不是分发
**API 并发上限**: 5 个 agent 并行

### 🔴 绝对禁止：计数器标记 [🤖x/5] 替代回复内容

**这是最高级别硬性规则，不可违反，违反 = INC 事故。**

- 每次回复**必须**先写完整的回复内容，然后空一行，最后一行写计数器。
- 如果回复只有计数器 `[🤖x/5]` 没有内容 = **最高级别错误 = INC 事故**
- 绝对禁止：用计数器代替正常回复（如本该说「收到，正在处理」却只发 `[🤖3/5]`）
- 绝对禁止：计数器前面没有任何实质性文字

> **🛡️ 自动防护**：`message-guard` 系统钩子会在消息发送前自动检测并修复「只有计数器」的回复。但这是安全网，不能替代你自身的正确行为。违反 = 仍记为 INC。

### 🔴🔴 每次回复底部必须添加 Agent 状态标记

格式：
```
[🤖x/5]
```

`x` = 当前空闲 agent 数量 = **5 - 当前活跃子代理数**

### 🔴🔴 硬性规则：计数器必须附加在正常回复内容末尾

**计数器标记绝对不能替代回复内容。这是最高优先级规则，违反 = INC 事故。**

#### 正确格式（✅ 必须遵守）

```
[完整回复内容]

[🤖x/5]
```

**结构要求**：
1. 先写完整的回复内容（确认、任务描述、结果、建议等）
2. 空一行
3. 最后一行写计数器标记 `[🤖x/5]` 或 `[🤖0/5 已满]`

#### 禁止行为（❌ 违反 = INC 事故）

1. ❌ **只发计数器不发内容** — 回复中只有 `[🤖x/5]`，没有任何实质内容
2. ❌ **计数器替代回复** — 本该说「收到，正在处理」却只发了 `[🤖3/5]`
3. ❌ **计数器混在内容中间** — 计数器必须单独一行放在末尾
4. ❌ **计数器后还跟其他内容** — 计数器必须是最后一行

#### ⚠️ 严重错误示例（必须牢记）

```
❌ ❌ ❌ 严重错误：只发计数器
[🤖4/5]
→ 用户完全不知道发生了什么，没有收到任何确认或信息
→ 违反 P0 规则 → INC 事故

✅ 正确：完整回复 + 计数器
收到，已分发任务到子代理，完成后汇报。

[🤖4/5]
```

#### 示例对比

```
❌ 错误 — 只有计数器，用户不知道发生了什么
[🤖4/5]

❌ 错误 — 计数器在内容中间
收到 [🤖4/5] 正在处理。

❌ 错误 — 计数器后还有内容
[🤖4/5]
有问题随时问我。

✅ 正确 — 完整回复 + 空行 + 计数器
收到，正在处理考勤数据分析，预计 3 分钟完成。

[🤖4/5]

✅ 正确 — 纯确认也要用完整句子
收到，已完成。

[🤖4/5]

✅ 正确 — 结果汇报 + 计数器
✅ 本周考勤分析完成！

📊 数据概览：
- 出勤率：95%
- 迟到：3 人次

[🤖3/5]

✅ 正确 — 全满排队状态
[🤖0/5 已满] 当前任务排队中，预计 2 分钟后有空位。
```

### 规则

| 状态 | 标记格式 | 主会话行为 |
|------|----------|------------|
| **有空闲** (x > 0) | `[🤖x/5]` | 优先 spawn 子代理，主会话不下场 |
| **全满** (x = 0) | `[🤖0/5 已满]` | 新任务排队，回复用户说明排队情况 |

### 关键提醒

- 每次生成回复时必须检查：我的回复有实质内容吗？还是只有计数器？**只有计数器 = INC 事故**
- 如果当前没有实质内容需要回复（比如纯确认），也必须用完整句子 + 计数器，**绝对禁止只发计数器**
- x > 0 时主会话自己执行重任务 → 违规，记为 INC 事故

## 核心原则

**主会话 = 指挥官，不做具体工作**

```
┌─────────────────────────────────────────────┐
│           主会话（指挥官）                      │
│                                             │
│  接收用户请求 → 解析任务 → 分发到子代理         │
│  收集结果 → 汇总汇报 → 等待下一个指令          │
│                                             │
│  ❌ 不直接执行数据处理                        │
│  ❌ 不直接调用 exec 处理数据                   │
│  ❌ 不直接读取大文件                           │
│  ❌ 不在主会话做重复性工具调用                   │
└──────────────┬──────────────────────────────┘
               │ 分发
       ┌───────┼───────┐
       ▼       ▼       ▼
   子代理A   子代理B   子代理C
   (临时)    (临时)    (临时)
   数据处理  文件操作  分析计算
```

---

## 强制规则

### ✅ 必须通过子代理执行的任务

| 任务类型 | 示例 | 原因 |
|---------|------|------|
| 数据处理/分析 | 考勤统计、报表生成 | 工具调用多、耗时长 |
| 文件批量操作 | 批量创建 Agent、批量更新 | 重复操作、易卡 |
| 大文件读取 | >5000 字符的文档 | 消耗上下文 |
| 复杂计算 | 评分模型、排名 | CPU 密集 |
| 多步骤工作流 | 数据拉取→分析→写报告 | 工具链长 |
| 飞书文档更新 | 大文档 overwrite | 网络请求慢 |

### ✅ 主会话只做的任务

| 任务类型 | 示例 |
|---------|------|
| 接收用户指令 | 理解需求、拆解任务 |
| 分发给子代理 | sessions_spawn |
| 汇总结果 | 合并子代理输出 |
| 简单查询 | 版本检查、状态查看 |
| 紧急回复 | 用户追问时立即响应 |
| 谛听复盘 | 质量检查、事故记录 |

## 分发流程（含回复顺序）

```
用户发任务
    ↓
① 主会话：3 秒内回复用户「收到，正在处理」
    ↓
② 主会话：解析 + 拆解为子任务
    ↓
③ sessions_spawn (mode="run", runtime="subagent")
    ↓
④ 子代理：并行执行各自任务
    ↓
⑤ 子代理完成 → 自动汇报结果
    ↓
⑥ 主会话：收集所有结果 → 汇总 → 发送给用户
```

### 示例：处理考勤分析任务

```
❌ 错误模式（旧）：
主会话：feishu_sheet 读取 → 读取9个表 → 写分析脚本 → 
        执行脚本 → 生成报告 → 更新飞书文档
        （主会话卡死 40 分钟）

✅ 正确模式（指挥官）：
主会话：sessions_spawn("考勤数据分析，读取9人考勤表，生成报告")
        sessions_spawn("飞书文档更新，将报告写入指定文档")
子代理A: 并行执行数据处理
子代理B: 并行执行文档更新
主会话：等待完成 → 汇总 → 发送给用户
```

## 🎻 Orchestra 核心规则（2026-04-09 融入）

来源：Orchestra 架构设计文档，提炼为 7 条可执行规则。

---

### 规则 1：任务通知系统（Push-Based 完成汇报）

**触发条件**：子代理完成或被终止

**执行动作**：
- 子代理完成后 **自动返回结果**（push-based），主会话无需 poll 状态
- 主会话收到子代理结果后，**立即转发给用户**，不等待其他子代理
- 多子代理场景：每完成一个就汇报一个，不等全部完成

**标准汇报格式**：
```
✅ [子代理名] 已完成
- 产出：[具体成果]
- 耗时：[X 分钟]
- 异常：[无 / 简述]
```

**❌ 禁止**：
- ❌ 主会话轮询子代理状态（不 poll `subagents list`）
- ❌ 子代理完成后主会话沉默（必须立即转发）
- ❌ 等所有子代理完成才统一汇报（逐个汇报更快）

---

### 规则 2：读写分离（ReadWorker / WriteWorker）

**触发条件**：任务涉及文件读取 + 写入操作

**执行动作**：
- **ReadWorker**（只读子代理）：用于分析、调研、信息收集
  - 可用工具：read, web_search, web_fetch, exec（只读命令）
  - 禁止：write, edit, 任何修改操作
- **WriteWorker**（写入子代理）：用于文件创建/修改
  - 可用工具：write, edit, feishu_update_doc 等写入工具
  - 禁止：exec 数据处理、大文件读取

**主会话判断表**：
| 任务类型 | 使用 | 示例 |
|---------|------|------|
| 仅读取/分析 | ReadWorker | 读取代码库结构、分析数据 |
| 仅写入/修改 | WriteWorker | 创建文档、修改配置 |
| 先读后写 | ReadWorker → WriteWorker | 分析后写报告 |
| 读+写交织 | 单个完整子代理 | 简单文件的读写修改 |

**优势**：只读子代理可安全并行（不会互相覆盖），写入子代理串行执行。

---

### 规则 3：灵活恢复（子代理失败处理）

**触发条件**：子代理执行失败 / 超时 / 产出异常

**执行动作**：

| 失败类型 | 处理方式 |
|---------|---------|
| **轻度失败**（部分完成） | 基于已有结果继续，不重新 spawn |
| **中度失败**（完全失败但原因明确） | spawn 新子代理，task 中包含失败原因和修正指示 |
| **重度失败**（连续 2 次失败） | 主会话直接介入，向用户说明问题并请求指示 |

**重试模板**：
```javascript
sessions_spawn({
  task: "原任务：[原任务描述]。上次失败原因：[具体原因]。请避免：[修正指示]",
  mode: "run",
  runtime: "subagent",
  label: "[原标签] - 重试"
})
```

**❌ 禁止**：
- ❌ 子代理失败后主会话沉默不处理
- ❌ 盲目重试不分析原因
- ❌ 超过 2 次重试仍不告知用户

---

### 规则 4：Continue vs Spawn 决策

**触发条件**：需要执行后续任务时

**决策规则**：

| 条件 | 决策 | 原因 |
|------|------|------|
| 子代理还需要继续对话/追问 | **Continue**（同一子代理） | 保留上下文 |
| 子代理已完成，需要新任务 | **Spawn**（新子代理） | 上下文干净 |
| 任务依赖上一个子代理的产出 | **Spawn**，在 task 中传递结果 | 解耦 |
| 同一任务的多个迭代步骤 | **Continue** | 上下文连贯 |
| 完全独立的新任务 | **Spawn** | 并行执行 |

**核心原则**：
- ✅ 任务间无依赖 → 并行 Spawn
- ✅ 任务间有依赖 → 串行 Spawn（上一个的产出作为下一个的输入）
- ✅ 同一任务需要多轮对话 → Continue
- ❌ 依赖关系不明确 → 先 Spawn 一个探索，再决定后续

---

### 规则 5：Scratchpad 知识共享

**触发条件**：多个子代理需要共享信息

**执行动作**：
- 主会话创建共享文件 `scratchpad/[任务名].md`，写入：
  - 任务目标
  - 已知约束
  - 子代理分工
  - 已完成的结果
- 子代理在 task 中指定读取该文件
- 子代理完成后将关键产出追加到 scratchpad
- 主会话在所有子代理完成后清理 scratchpad 文件

**Scratchpad 格式**：
```markdown
# [任务名] - Scratchpad
## 目标
[一句话描述]

## 已知约束
- [约束 1]
- [约束 2]

## 子代理分工
- Agent A: [任务]
- Agent B: [任务]

## Agent A 产出
[结果]

## Agent B 产出
[结果]

## 最终汇总
[由主会话填写]
```

**使用场景**：
- 多子代理协作的复杂项目（>3 个子代理）
- 需要阶段性信息传递的流水线任务
- 子代理产出需要被后续子代理引用

---

### 规则 6：工具权限控制

**触发条件**：spawn 子代理时

**执行动作**：
在 task 描述中明确指定可用/不可用工具范围：

```javascript
sessions_spawn({
  task: "[任务描述]。可用工具：read, web_search。禁止使用：exec, write, message",
  mode: "run",
  runtime: "subagent",
  label: "[标签]"
})
```

**默认权限矩阵**：
| 子代理类型 | 允许 | 禁止 |
|-----------|------|------|
| 调研型 | read, web_search, web_fetch | write, edit, exec, message |
| 分析型 | read, exec（只读命令） | write, edit, message |
| 写入型 | write, edit, feishu_update_doc | exec（数据处理）, message |
| 全能型 | 全部工具 | 无（需主会话明确授权） |

**❌ 硬禁止**（任何子代理都不可用）：
- ❌ 删除生产环境文件
- ❌ 发送邮件/消息（除非主会话明确指定收件人）
- ❌ 修改 SOUL.md / AGENTS.md / CORE-RULES.md

---

### 规则 7：四阶段工作流（复杂任务标准流程）

**触发条件**：复杂任务（>3 个子代理、涉及多步骤）

**四个阶段**：

```
阶段 1：分析分解
  主会话：理解任务 → 拆解子任务 → 确定依赖关系
  产出：子任务清单 + 依赖图

阶段 2：信息收集（并行 ReadWorkers）
  主会话：并行 spawn 多个 ReadWorker
  产出：各子代理产出 → 写入 Scratchpad

阶段 3：执行产出（WriteWorkers）
  主会话：基于阶段 2 的结果，spawn WriteWorkers
  产出：最终交付物

阶段 4：验证汇总
  主会话：收集所有结果 → 质量检查 → 汇总汇报
  产出：给用户的最终报告
```

**每个阶段的 gate**：
- 阶段 1 → 2：子任务清单已确认
- 阶段 2 → 3：关键信息已收集完毕
- 阶段 3 → 4：所有交付物已完成
- 阶段 4 → 用户：质量检查通过

**简单任务跳过**：≤2 个子代理的任务可直接进入阶段 3，跳过阶段 1-2。

### 规则 8：讨论 vs 指令的区分

**触发条件**：主会话需要决定与子代理的交互模式

**区分规则**：

| 场景 | 模式 | 方式 | 示例 |
|------|------|------|------|
| 需要子代理给出分析/建议/方案 | **讨论模式** | task 中说明「请分析后回复你的建议」 | 「分析这份数据，告诉我可能的原因」 |
| 需要子代理直接执行并产出结果 | **指令模式** | task 中明确输出格式和路径 | 「将分析结果写入 /tmp/report.md」 |
| 不确定方案，需要多轮迭代 | **Continue 模式** | 不立即 spawn 新代理，在同一子代理内多轮对话 | 「基于这个方案，进一步细化第三步」 |
| 方案已确认，需要落地执行 | **指令模式 + Spawn** | spawn 新子代理，task 中包含最终方案 | 「按方案 A 执行，输出到指定路径」 |

**核心原则**：
- ✅ 探索性任务 → 讨论模式（子代理输出建议，主会话决策）
- ✅ 执行性任务 → 指令模式（子代理直接产出结果）
- ✅ 讨论后确认方案 → 新 spawn 指令型子代理执行（避免上下文污染）
- ❌ 讨论和指令混在同一个子代理中（会导致产出模糊）

**决策流程**：
```
用户发任务
    ↓
任务类型是「分析/调研/给建议」吗？
    ↓
┌──────✅ 是 ──────────┐
│ 讨论模式              │
│ task:「请分析...给出建议」│
│ 子代理输出建议          │
│ 主会话决策              │
│ ↓                     │
│ 方案确认后              │
│ spawn 新指令型子代理     │
└───────────────────────┘
    ↓ ❌ 否
指令模式
任务明确 → 直接 spawn 执行
```

### 规则 9：飞书文档同步规则

**触发条件**：任务涉及飞书文档的读写/更新/创建

**执行动作**：

**读取飞书文档**：
- 使用 `feishu_fetch_doc` 工具
- 文档 >5000 字符 → 必须通过 ReadWorker 子代理读取
- 多文档读取 → 并行 spawn 多个 ReadWorker

**写入/更新飞书文档**：
- 使用 `feishu_update_doc` 或 `feishu_create_doc` 工具
- 必须通过 WriteWorker 子代理执行
- 大文档更新（>2000 字符）→ 使用 `task_id` 异步模式，主会话不等待

**创建飞书文档**：
- 使用 `feishu_create_doc` 工具
- 需要指定位置时，在 task 中传入 `folder_token` 或 `wiki_space`

**飞书文档操作检查清单**：
- [ ] 文档 token/URL 已正确传递到子代理 task 中
- [ ] 更新模式（overwrite/append/replace_all）已明确指定
- [ ] 大文档更新使用异步 task_id 模式
- [ ] 子代理完成后主会话验证文档是否更新成功

**与 feishu-update-doc Skill 的关系**：
- 具体的更新模式（7 种）详见 `skills/feishu-update-doc/SKILL.md`
- 具体的创建流程详见 `skills/feishu-create-doc/SKILL.md`
- 指挥官模式只规定「谁来做」（子代理），具体「怎么做」由对应 Skill 决定

**❌ 禁止**：
- ❌ 主会话直接执行 feishu_update_doc（除非是简单的 append 一行）
- ❌ 子代理更新文档后不验证结果
- ❌ 不传 task_id 同步等待大文档更新

**主会话接任务后的 5 秒检查**：
```

5️⃣ 子代理完成 → 立即转发给用户
```

**子代理类型速查**：
```
调研/分析 → ReadWorker  （只读，可并行）
写入/修改 → WriteWorker （写入，需串行）
探索/建议 → 讨论模式     （输出建议供决策）
执行/产出 → 指令模式     （直接产出结果）
```

**失败处理速查**：
```
部分完成 → 基于已有结果继续
完全失败 → 分析原因 → spawn 新子代理（带修正指示）
连续 2 次失败 → 主会话介入，向用户说明
```

**四阶段工作流速查**：
```
分析分解 → 信息收集（并行） → 执行产出 → 验证汇总
  ↓            ↓                  ↓           ↓
子任务清单   ReadWorkers      WriteWorkers  最终报告
```

## 🔌 与 Orchestra（奥凯斯特拉）的关系与互斥规则

**来源**: 共存分析 2026-04-10
**详细分析**: `orchestra/COEXISTENCE-ANALYSIS.md`

### 核心认知

指挥官模式 **不是** Orchestra 的替代品，而是 **Orchestra 理念的 LLM 可执行版本**。

本 SKILL.md 中的规则 1-10 已经从 Orchestra 架构设计文档中提炼而来，包含：
- 任务通知系统、读写分离、灵活恢复
- Continue vs Spawn 决策矩阵
- Scratchpad 知识共享、工具权限控制
- 四阶段工作流、讨论 vs 指令区分
- 飞书文档同步

**指挥官模式 = Orchestra 的灵魂 + LLM 行为规则**

### 🔴 互斥规则（最高优先级）

| 规则 | 要求 | 原因 |
|------|------|------|
| **不运行 gateway.js** | 主会话禁止执行 `node orchestra/gateway.js` | 双重调度 = 子代理重复 |
| **不执行 Orchestra 代码** | 主会话不运行 orchestra/ 下的 .js/.ts 调度代码 | 指挥官模式已覆盖所有调度逻辑 |
| **统一 Scratchpad 路径** | 子代理协作统一使用 `scratchpad/[任务名].md` | Orchestra 的 JSON scratchpad 与 Markdown 不兼容 |
| **Orchestra 仅作参考** | orchestra/ 目录作为设计理念参考，不作为运行时 | 指挥官模式 = 运行时，Orchestra = 设计文档 |

### ✅ 允许的使用方式

```text
✅ 日常使用：主会话遵循指挥官模式规则，spawn 子代理
✅ 设计参考：阅读 Orchestra 代码学习架构设计
✅ 独立运行：用户明确要求时，单独运行 node orchestra/gateway.js（不混用）
```

### ❌ 禁止的使用方式

```text
❌ 主会话同时遵循指挥官模式 + 运行 gateway.js → 双重调度
❌ 主会话读取 Orchestra DecisionMatrix 后，又用自己的规则做决策 → 规则冲突
❌ 子代理同时写入 orchestra/temp/scratchpad/ 和 scratchpad/ → 数据孤岛
```

### Scratchpad 统一规则

```
workspace/scratchpad/[任务名].md    ← 唯一使用的协作路径
workspace/orchestra/temp/           ← Orchestra 内部数据（仅独立运行时使用）
```
