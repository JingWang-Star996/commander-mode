# 指挥官模式 (Commander Mode) — OpenClaw 技能

> **解决"AI 沉默执行"问题**：给 AI 一个复杂任务后，它沉默好几分钟才回复。指挥官模式解决这个问题。

**已在生产环境验证。** 不是 OpenClaw 官方功能，是来自真实生产环境的社区实践。

---

## 解决的问题

指挥官模式之前：

```
用户："帮我分析这个月的考勤数据，生成报告"
         ⏳ ... 沉默 40 秒 ...
         ⏳ ... 还是没有回复 ...
         ⏳ ... AI 正在执行 40+ 次工具调用 ...
         ✅ 终于："完成了！这是报告。"
```

LLM 的默认行为是 **思考/执行 → 输出**。用户感知为 **等了很久**。

## 解决方案

指挥官模式强制一条简单规则：**先回复，再分发。**

```
用户发任务
    ↓（3 秒内）
① 主会话：立即回复用户
    "收到，正在处理，预计 3 分钟完成。"
    ↓（同时）
② 主会话：spawn 子代理执行重活
    ↓
③ 子代理：并行执行
    ↓
④ 结果自动推回 → 主会话转发给用户
```

**效果：响应时间从 ~40 秒降到 ~3 秒。** 主会话工具调用从 40+ 次降到 ≤3 次。

---

## 核心机制

### 1. 先回复再分发（P0）

主会话 **必须** 在用户发消息后先回复，再做任何工具调用。这是最重要的规则。

### 2. Agent 空闲计数器

每条回复底部附带计数器 `[🤖x/5]`，显示可用子代理槽位（共 5 个并发）。防止主会话在有空闲 agent 时自己干活。

### 3. 子代理并行分发

独立任务并行执行，依赖任务串行传递结果。主会话不干重活。

---

## 包含的规则

| 规则 | 作用 |
|------|------|
| **先回复 (P0)** | 3 秒内回复用户，在任何工具调用之前 |
| **Agent 计数器** | `[🤖x/5]` 追踪 — 防止主会话忘记分发 |
| **推送式完成** | 子代理自动汇报，主会话立即转发（不轮询） |
| **读写分离** | ReadWorker（只读，可并行）vs WriteWorker（串行） |
| **灵活恢复** | 子代理失败时的优雅处理和重试逻辑 |
| **Continue vs Spawn** | 决策指南：何时继续对话，何时 spawn 新代理 |
| **Scratchpad 共享** | 多子代理协作的共享状态文件 |
| **工具权限** | 每种子代理类型有明确的允许/禁止工具列表 |
| **四阶段工作流** | 分析 → 收集 → 执行 → 验证（复杂任务标准流程） |
| **讨论 vs 指令** | 区分探索性讨论和执行指令 |
| **飞书文档同步** | 通过子代理读写飞书文档的规则 |

## 安装方法

1. 复制 `SKILL.md` 到你的 OpenClaw skills 目录：
   ```bash
   cp SKILL.md ~/.openclaw/workspace/skills/commander-mode/SKILL.md
   ```
2. 重启 OpenClaw（或等待下次技能自动加载）。
3. 技能会在匹配的任务描述时自动加载。

## 快速参考

```
接到任务？
  1️⃣ 先回复用户（3 秒内）
  2️⃣ 简单？→ 自己做
     复杂？→ spawn 子代理
  3️⃣ 有依赖？→ 串行 | 无依赖？→ 并行
  4️⃣ 子代理完成 → 立即转发
```

## 适用场景

- **复杂数据处理**（考勤分析、留存指标）
- **批量文档操作**（批量创建、批量更新飞书文档）
- **多步骤工作流**（拉数据 → 分析 → 写报告 → 发布）
- **任何需要 >3 次工具调用的任务**

## 不适用的场景

- 简单查询（"现在几点了？"）
- 单次工具调用的任务
- 转发消息

---

## 为什么需要这个

OpenClaw 内置了子代理支持，但默认不会强制 **先回复的纪律**。没有它，LLM 还是会走"执行后再回复"的老路 — 导致长时间沉默。指挥官模式用明确的规则和计数器作为安全网，弥补了这个缺口。

打个比方：**OpenClaw 提供了管道，指挥官模式提供了纪律。**
